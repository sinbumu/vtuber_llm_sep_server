# CONTEXT_COMPACTION_ANALYSIS_TASK.md

## 목적

현재 프로젝트의 대화 컨텍스트 관리 구조를 분석하고,  
장시간 대화 시에도 **페르소나와 중요한 맥락은 유지하면서**,  
불필요하게 누적되는 대화 히스토리는 **주기적으로 축약/정리**할 수 있는 구조를 제안한다.

이번 작업의 목표는 **즉시 구현이 아니라 사전 조사 및 설계 정리**다.

즉, 아래를 명확히 파악하고 싶다.

1. 현재 프로젝트가 대화 히스토리를 어떤 방식으로 저장/주입하는지
2. 어느 지점에 컨텍스트 축약 로직을 넣는 것이 가장 안전한지
3. 어떤 정보는 반드시 유지해야 하고, 어떤 정보는 요약/폐기 가능한지
4. MVP 수준에서 가장 구현 부담이 적은 방식이 무엇인지

---

## 배경

현재 프로젝트는 기존 오픈소스 VTuber LLM 구조를 기반으로 하고 있으며,  
대화 히스토리를 사실상 **무한 누적**하는 구조에 가깝다.

이 구조는 장시간 대화 시 다음 문제가 생길 수 있다.

- 컨텍스트 길이 초과
- 응답 품질 저하
- 불필요한 과거 대화까지 계속 입력에 포함
- 중요 정보와 잡담이 동일 가중치로 누적
- 장기적으로 비용/속도/안정성 문제

우리가 원하는 방향은 다음과 같다.

- 캐릭터 페르소나, 핵심 사용자 설정, 중요한 대화 맥락은 유지
- 일반 잡담, 반복되는 짧은 대화, 중요도가 낮은 로그는 요약 또는 축약
- 구조적으로 너무 복잡한 장기 기억 시스템보다는, 현재 코드에 맞는 **현실적인 경량화 방식** 우선 검토

---

## 조사 목표

다음 질문들에 답할 수 있도록 프로젝트 구조를 분석하라.

### 1. 현재 대화 컨텍스트는 어디서 어떻게 쌓이는가?
확인할 것:
- 히스토리 저장 위치
- 메모리 객체 구조
- 모델 호출 직전 messages 구성 로직
- system prompt / persona prompt / tool prompt 주입 구조
- conversation history가 어떤 형태로 누적되는지

### 2. 컨텍스트 축약을 넣을 수 있는 가장 적절한 지점은 어디인가?
후보 예시:
- history 저장 직전
- memory 로딩 시점
- model 호출 직전 messages 빌드 시점
- 별도 compaction step
- 주기적 background-like compaction (실제 비동기 작업이 아니더라도 논리적 단계)

### 3. 어떤 정보는 반드시 유지해야 하는가?
최소한 아래는 분리해서 고려하라:
- 캐릭터 페르소나
- 캐릭터 설정 / emotionMap / 메이트 정보
- 사용자 관련 핵심 정보
- 현재 세션에서 이어져야 하는 중요한 맥락
- 장기적으로 남길 가치가 낮은 일반 잡담

### 4. 어떤 방식이 현재 코드에 가장 적합한가?
후보 방식 예시:
- sliding window
- rolling summary
- summary + recent window 혼합
- 중요 정보 추출 + 나머지 삭제
- history 전체 저장 + model 입력만 경량화
- 기타 현재 구조에 적합한 방법

단, 무리한 구조 변경보다는  
**현재 코드 흐름을 크게 깨지 않는 방식**을 우선적으로 제안하라.

## 전제 조건

- 현재 프로젝트의 기존 기능은 최대한 유지하고 싶다.
- 이번 단계에서는 장기 기억, 벡터 DB, RAG 같은 무거운 구조는 우선순위가 아니다.
- 우선은 **대화 컨텍스트를 가볍게 유지하는 현실적인 방법**이 목표다.
- 페르소나와 중요한 설정은 절대 손상되면 안 된다.
- 히스토리 원본은 가능하면 보존하고, 모델에 넣는 입력만 경량화하는 방향도 검토 대상이다.

## 조사 결과에 포함해야 할 항목

### 1. 현재 구조 요약
- 히스토리 누적 방식
- 컨텍스트 주입 방식
- 페르소나/설정 반영 방식
- 현재 구조의 문제점

### 2. 축약 대상 분류
아래처럼 분류하라.

- 절대 유지해야 하는 정보
- 최근 몇 턴만 유지하면 되는 정보
- 주기적으로 요약 가능한 정보
- 그냥 버려도 되는 정보

### 3. 삽입 지점 제안
코드상 어디에 축약 로직을 넣는 것이 좋은지 제안하라.

예:
- `_to_messages()` 직전
- `set_memory_from_history()` 이후
- `chat_history_manager` 계층
- 별도 `compact_context()` 함수 도입
- 기타

각 제안에 대해:
- 장점
- 위험성
- 구현 난이도
를 적어라.

### 4. 구현 방식 후보 비교
최소 아래 3가지는 비교하라.

#### 안 A. Recent window만 유지
- 최근 N개 대화만 model input에 사용
- 나머지는 원본 history에만 남김

#### 안 B. Summary + recent window
- 오래된 대화는 summary로 압축
- 최근 대화만 원문 유지

#### 안 C. 중요 정보 분리 + recent window
- 핵심 정보는 별도 memory block
- 최근 대화만 원문 유지

각 안에 대해:
- 현재 코드 적합성
- 구현 난이도
- 품질 기대치
- 추후 확장성
비교하라.

### 5. MVP 권장안
- 현재 프로젝트에 가장 적합한 1차 구현안을 추천하라.
- 왜 그 안이 가장 현실적인지 설명하라.

### 6. 차후 확장 방향
- 2차 단계에서 어떤 식으로 고도화할 수 있는지
- 예: summary 품질 향상, 중요 정보 구조화, runtime compaction 정책 등
