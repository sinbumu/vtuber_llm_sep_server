# SCREEN_CAMERA_CHAT_SUPPORT_TASK.md

## 목적

기존 오픈소스 코드베이스에 있었던 화면 공유 / 카메라 입력 기능을 참고하여,  
현재 분리된 LLM Python 서버의 `/chat` API 및 관련 채팅 경로에서  
**정지 이미지 기반 입력(screen snapshot / camera snapshot)** 을 다시 지원하도록 구조를 분석하고 구현안을 제안 및 반영한다.

이번 작업의 목표는 **실시간 영상 스트리밍이 아니라**,  
기존 오픈소스 수준의 **단일 이미지 또는 주기적 스냅샷 전달** 방식 지원이다.

---

## 작업 목표

다음을 달성하는 것을 목표로 한다.

1. 기존 오픈소스 코드에서 화면 공유 / 카메라 입력이 어떻게 구현되어 있었는지 추적
2. 현재 분리 서버 구조에서 해당 기능을 최소 변경으로 다시 붙일 수 있는지 분석
3. `/chat` 또는 현재 사용 중인 채팅 API 경로에서 이미지 입력을 받을 수 있도록 설계
4. Unity 앱이 나중에 정지 이미지 파일 또는 base64 이미지를 쉽게 전달할 수 있도록 입력 구조를 정의
5. 기존 텍스트 채팅 기능은 유지하고, 이미지 입력은 **옵션 기능**으로 추가

---

## 전제 조건

- 실시간 영상 스트림은 지원 대상이 아님
- 화면 공유 / 카메라 기능은 “정지 이미지 입력” 방식만 검토
- Unity 앱이 추후 아래 방식 중 하나로 이미지를 전달할 수 있어야 함
  - 파일 경로
  - multipart upload
  - base64 string
- 현재 서버 구조는 최대한 유지
- 기존 텍스트 전용 `/chat` API를 깨지 않는 방식 우선
- 기능이 불안정하거나 구조적으로 과한 경우, Phase 1에서는 “기초 지원만” 넣어도 됨

## 반드시 확인해야 할 핵심 질문

### 1. 기존 오픈소스에서 이미지 입력은 어떤 구조였는가

다음을 추적하라.

- 화면 공유 / 카메라 관련 코드 파일 위치
- 이미지 캡처 또는 입력 객체가 어디서 생성되는지
- LLM 호출 직전 messages에 이미지가 어떻게 포함되는지
- 어떤 provider/model에서 이미지 입력을 지원하도록 되어 있었는지
- 텍스트 + 이미지 혼합 입력이 어떻게 구성되었는지
- OpenAI compatible / Gemini / 기타 멀티모달 provider 대응 여부

---

### 2. 현재 분리 서버에서 이미지 입력이 가능한 지점은 어디인가

다음을 확인하라.

- 현재 `/chat` 및 `/v1/chat`, `/v1/ws/chat` 등 실제 채팅 엔드포인트 구조
- 요청 payload schema
- `BatchInput`, `messages`, `input_data` 또는 유사 객체 구조
- `_to_messages()` 또는 model input builder가 이미지 content를 받을 수 있는지
- provider adapter가 image content를 처리할 수 있는지

---

### 3. 어떤 API 형태가 가장 현실적인가

아래 방식들을 비교 검토하라.

#### 안 A. `/chat` JSON payload에 image_path 추가
예:
```json
{
  "conf_uid": "...",
  "history_uid": "...",
  "text": "이 화면 보고 설명해줘",
  "image_path": "C:/.../snapshot.png"
}
````

#### 안 B. `/chat` JSON payload에 image_base64 추가

예:

```json
{
  "conf_uid": "...",
  "history_uid": "...",
  "text": "이 화면 보고 설명해줘",
  "image_base64": "...."
}
```

#### 안 C. multipart/form-data 방식

* text + file upload

각 안에 대해 다음을 비교하라.

* 구현 난이도
* Unity 연동 편의성
* 보안/안전성
* 서버 내부 처리 단순성
* 추후 웹 테스트 편의성

---

### 4. 현재 구조에서 최소 개조 지점은 어디인가

다음을 확인하라.

* 입력 모델(Pydantic schema 등) 수정 위치
* API handler 수정 위치
* 이미지 전처리/검증 로직 삽입 위치
* LLM messages builder 수정 위치
* provider별 멀티모달 처리 분기 위치
* history 저장 시 이미지 입력을 어떻게 남길지

## 분석 대상에서 우선적으로 봐야 할 코드 영역

다음 영역을 우선 추적하라.

1. 현재 채팅 API 엔트리포인트
2. request schema / payload model 정의
3. `BatchInput` 또는 유사 입력 객체
4. `_to_messages()` 또는 model input builder
5. stateless_llm / provider adapter / completion 호출 코드
6. 과거 오픈소스의 화면 공유 / 카메라 처리 코드
7. history 저장/로드 구조
8. 이미지 content type / base64 처리 유틸이 이미 있는지 여부

---

## 구현 시 반드시 지켜야 할 원칙

1. 기존 텍스트 전용 채팅은 그대로 동작해야 한다
2. 이미지 입력은 optional field로 추가한다
3. 실시간 스트리밍 구현은 범위에서 제외한다
4. 기존 오픈소스에서 되던 것을 가능한 한 재사용하되, 현재 분리 서버 구조에 맞게 단순화한다
5. 불필요하게 TTS / OBS / UI 코드까지 다시 살리지 않는다
6. Unity가 나중에 붙이기 쉬운 방향을 우선한다

예시:

* messages builder에서 text + image content를 함께 구성
* history에는 이미지 입력 메타만 저장

* 이미지 + 텍스트 요청 정상 처리 확인
* 이미지 없는 요청 backward compatibility 확인
* 지원하지 않는 provider에서 graceful error 처리

## 특히 확인해야 할 정책

다음 항목은 구현 전에 명확히 분리해서 정리하라.

### A. 서버가 직접 캡처를 하는가, Unity가 캡처해서 보내는가

권장 기본 방향:

* Unity가 캡처하고 서버는 받기만 하는 구조 우선 검토

### B. history에는 무엇을 저장할 것인가

권장 검토:

* 원본 이미지 데이터는 저장하지 않고
* `image_attached: true`, `image_source: screen/camera`, `filename` 또는 요약 메타만 저장

### C. provider 미지원 시 처리

* 멀티모달 미지원 provider면 명확한 에러 반환
* 또는 이미지 무시 금지, 명시적으로 실패 처리
