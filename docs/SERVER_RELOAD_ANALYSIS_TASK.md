# SERVER_RELOAD_ANALYSIS_TASK.md

## 목적

현재 분리된 LLM Python 서버를 Unity 앱에서 제어 가능한 형태로 개조하기 위한 **사전 분석**을 수행한다.

핵심 목표는 다음 두 가지를 명확히 구분하는 것이다.

1. **서버 재시작이 반드시 필요한 설정 요소**
2. **서버 실행 중 설정 파일 재로드 또는 통신을 통해 갱신 가능한 요소**

최종적으로는 Unity 앱이 Python 서버 `.exe`와 동일 경로의 설정 파일을 관리하고,  
서버 실행 중에는 필요 시 설정 파일을 다시 읽어 반영하도록 개조 가능한지 판단한다.

---

## 전제 조건

- Unity 앱이 Python 서버를 외부 프로세스 형태(`.exe`)로 실행/관리할 예정
- Unity 앱은 서버와 HTTP / WebSocket 등으로 통신 가능
- Python 서버는 `.exe`와 같은 경로 또는 지정 경로의 설정 파일을 읽어 초기화됨
- 장기적으로는 “재시작 없는 설정 반영”을 지원하고 싶지만, 실제로 어려운 설정은 “재시작 필요”로 분리해도 됨
- 현재 서버의 전체 기능은 최대한 유지하는 방향을 우선 검토
- 분석 대상 AI는 **GPT Codex 계열 모델**을 사용한다고 가정

---

## 작업 범위

다음을 중심으로 현재 서버 구조를 분석한다.

1. 서버 시작 시점의 설정 로딩 구조
2. 설정값을 사용하는 객체/서비스 초기화 흐름
3. 런타임 재로드가 가능한 설정과 불가능한 설정의 분류
4. Unity 연동용 관리 API 추가 가능성
5. 최소 개조안 및 MVP 구현 방향 제안

## 반드시 확인해야 할 핵심 질문

### A. 현재 서버 초기화 구조 분석

다음을 확인하라.

1. 서버 프로세스 시작 시 어떤 설정 파일/환경변수를 읽는가
2. 설정을 읽고 나서 어떤 객체/서비스를 초기화하는가
3. 초기화 이후 메모리에 고정되는 요소는 무엇인가
4. LLM provider / model / base_url / api_key / persona / tool prompts 등이 언제 어떤 방식으로 주입되는가
5. TTS, 감정 처리, MCP, prompt loader, service context 등이 어떤 생명주기를 가지는가

---

### B. 설정 반영 시점 분류

다음 항목들을 각각 분류하라.

#### 1) 서버 재시작이 반드시 필요한 요소
예시:
- host / port 같은 프로세스 레벨 bind 정보
- 서버 시작 시점에만 생성되는 객체
- 외부 프로세스 / MCP / 플러그인 초기화 요소
- 재초기화 시 연결/상태가 깨지는 요소

#### 2) 런타임 재로드 가능성이 있는 요소
예시:
- provider / model / base_url / api_key
- persona_prompt
- tool_prompts
- temperature / max_tokens
- 기타 요청 단위로 바뀔 수 있는 설정

각 항목에 대해 아래를 함께 정리하라.

- 왜 재시작이 필요한지 또는 왜 런타임 변경이 가능한지
- 최소 개조 포인트가 어디인지
- 실제 코드 근거가 무엇인지

---

### C. 설정 파일 재로드 방식 가능 여부 분석

다음 시나리오를 기준으로 검토하라.

#### 시나리오
- Unity가 서버와 같은 경로의 설정 파일을 수정
- 서버가 `/admin/reload-config` 또는 유사 명령을 수신
- 서버가 설정 파일을 다시 읽고 적용 가능한 항목만 재초기화
- 적용 불가능한 항목은 “재시작 필요” 응답 반환

#### 확인할 것
1. 현재 구조에서 설정 파일 재로드 진입점을 어디에 두는 것이 적절한가
2. 설정 재로드 시 어떤 객체를 다시 생성해야 하는가
3. 기존 세션 / history / memory 와 충돌이 생기는가
4. 재로드 실패 시 롤백 또는 실패 응답 처리가 가능한가
5. concurrent request 중 재로드 시 안전성 문제가 있는가

---

### D. Unity 연동 관점에서 필요한 서버 API 제안

다음 API 또는 기능이 필요한지 검토하라.

#### 1. `GET /health`
- 서버 상태 확인

#### 2. `GET /admin/current-config`
- 현재 적용 중인 설정 조회

#### 3. `POST /admin/reload-config`
- 설정 파일 재로드 시도
- 성공 / 실패 / 재시작 필요 항목 반환

#### 4. `POST /admin/set-runtime-options`
- 일부 설정만 런타임 변경

각 API에 대해 아래를 정리하라.

- 필요한 이유
- 현재 코드에서 구현 난이도
- 보안 / 노출 범위 주의사항

## 판단 기준

분석 시 다음 기준을 우선 적용하라.

1. 현재 코드 변경을 최소화할 것
2. Unity와의 연동이 단순할 것
3. 런타임 재로드는 “안전한 항목만” 허용할 것
4. 불확실하면 재시작 필요로 분류할 것
5. 전체 기능 유지를 우선하되, 구조적으로 위험한 기능은 별도 분리할 것

---

## 금지 사항

1. 근거 없이 “가능할 것 같다” 식으로 추정하지 말 것
2. 실제 코드 흐름을 확인하지 않은 항목은 명확히 “미확인”으로 표기할 것
3. 구현 제안 시 반드시 코드 기준 근거 파일/함수 위치를 함께 제시할 것
4. 리팩토링을 전제로 답하지 말고, 현재 구조를 기준으로 현실적인 최소 개조안을 우선 제시할 것
